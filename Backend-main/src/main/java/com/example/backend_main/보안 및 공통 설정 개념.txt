####### 백엔드 보안 및 관리자 시스템 구축 가이드!! ########

1. 실제 시민 : Entity (실제 데이터의 본체)
    엔티티는 데이터베이스라는 '도시'에 실제로 살고 있는 [시민]과 같습니다.
    자바 세상의 클래스와 DB 세상의 테이블을 1:1로 연결해주는 아주 특별한 존재입니다.

- 왜 시민인가요?
: 코드로 만든 객체가 단순히 메모리에 떠돌다 사라지는 게 아니라,
    DB라는 금고(테이블)에 이름표를 달고 영구적으로 저장이 되기 때문에

- @Entity & @Table
: 해당 클래스가 단순한 도구가 아니라 DB와 연결된 [진짜 시민]임을 선언하고,
    [TB_USER]라는 특정 구역(테이블)에 살 것임을 지정합니다.

- 보안 장치(Protected 생성자)
: @NoArgsConstructor(access = AccessLevel.PROTECTED)는 JPA(시스템) 외에 일반 개발자가
    함부로 알맹이 없는 빈 시민 객체를 만들지 마라!!! 라고 입구를 좁혀둔 보안 장치입니다.


2. 배달원 : DTO(Data Transfer Object)
    DTO는 성 밖(리액트)과 성 안(스프링)을 오가는 택배 배달원입니다.
    시민(Entity)이 직접 성 밖으로 나가면 개인 정보가 노출될 위험이 크기 때문에, 꼭 필요한 정보만 바구니(DTO)에 담습니다.

- 왜 DTO를 사용하나요?
1) 보안 : 시민의 주민번호나 비밀번호를 성 밖으로 그대로 내보내지 않기 위해서입니다.
2) 검증 : 배달원이 물건을 받기 전, 서류 양식이 맞는지(이메일 형식, 글자 수 등)를 미리 검사할 수 있습니다.

- 이중 유효성 검사 (Validation)
: 리액트에서 한 번 검사하지만, 백엔드에서도 @NotBlank, @Email 등을 사용하여 [서류 미비]인 데이터가
    성 안으로 들어오지 못하게 원천 차단합니다.


3. 표준 식판(객체) : ResultVO (표준 응답 양식)
    ResultVO는 성을 찾아온 손님(프론트엔드)에게 음식을 내어주는 [표준 식판]입니다.
    메뉴타 무엇이든(회원 목록/성공 메시지 등) 항상 같은 자리에 같은 반찬이 놓여야 손님이 당황하지 않습니다.

- 구성 요소
1) Success : 이번 요청이 성공/실패의 여부를 알려주는 신호등입니다.
2) Message : [로그인 성공]이나 [비밀번호가 틀렸습니다]처럼 사용자에게 보여줄 친절한 설명서입니다.
3) Data : 실제 손님이 주문한 음식(회원 정보, 토큰 등)이 담기는 만능 주머니입니다.

- 핵심!!!!!!
: 프론트 엔드에서 이 식판 구조만 알면 어떤 API든 쉽게 처리할 수 있습니다!!


4. 창고 관리자 : Repository (DB 입출력 창구)
    레포지토리(Repository)는 DB라는 거대한 창고에서 시민을 찾거나 새로 등록해주는 [창고 관리자]입니다.

- JpaRepository 상속
: 자바 표준 규격을 상속받는 것만으로 저장(save)/ 찾아와라(findById)같은 복잡한 SQL 명령어를 직접 짜지 않아도
    관리자가 알아서 처리해줍니다.

- Optional의 마법
: 시민을 찾으러 갔는데 없을 경우, [없음]이라는 상태를 안전하게 전달하여,
    프로그램이 갑자기 멈추는(Null 에러) 현상을 방지합니다.


5. 비밀번호의 절구질 : BCrypt (단방향 해시)
비밀번호는 성주(관리자)도 알 수 없어야 합니다.
그래서 복호화(다시 풀기)가  불가능한 단방향 암호화를 사용합니다.

- BCrypt의 원리
: 비밀번호를 절구에 넣고 아주 곱게 빻는 것과 같습니다.
빻은 가루를 보고 원래의 비빔밥 재료를 알아낼 수 없듯이, 저장된 해시값을 보고 원래 비밀번호를 찾는 것은 불가능합니다.

- Salt (소금치기)
: 같은 비밀번호라도 뿌리는 소금(Salt)에 따라 결과물이 다르게 나옵니다.
해커가 미리 계산해둔 정답지를 쓰지 못하게 방해하는 기술입니다.

- 핵심 코드 한 줄 설명
코드 : BCrypt.hashpw(password, BCrypt.gensalt())
BCrypt.gensalt(): : 매번 다른 랜덤한 소금 준비하기
BCrypt.hashpw(...) : 준비한 소금과 비번을 섞어 아주 복잡한 문자열로 빻아버리기


6. 개인정보의 마법 금고 : AES-256 (양방향 암호화)
이메일이나 전화번호는 나중에 서비스(이메일 발송 등)를 위해 다시 읽어야 할 때가 있습니다.
그래서 열쇠로 잠그고 다시 열 수 있는 양방향 암호화를 사용합니다.

- AES-256
: 현재 전 세계적으로 가장 안전하다고 인정받는 금고 규격입니다.
    열쇠 길이가 256비트(32자)로 매우 길어 해킹이 거의 불가능합니다.

- IV (Initial Vector, 흔들기 규칙)
: 똑같은 이메일이라도 금고를 흔드는 규칙(IV)을 매번 다르게 하여, 암호화된 결과물이 매번 다르게 나오게 합니다.

- 암호화 과정(encrypt) 한 줄 설명
1) SecureRandom().nextBytes(iv) : 금고를 흔들 랜덤한 규칙(16 바이트)을 생성합니다.
2) cipher.init(Cipher.ENCRYPT_MODE, keySpec, ivSpec) : 기계에 열쇠를 꽂고 [잠금 모드] 스위치를 올립니다.
3) Base64.getEncoder().encodeToString(...) : 암호화된 깨진 글자를 DB에 저장하기 좋게 깨끗한 문자열로 포장하기


7. 디지털 신분증 :JWT(Json Web Token)
사용자가 로그인에 성공하면, 매번 아이디/비번을 물어보는 대신 성 안을 자유롭게 다닐 수 있는 [디지털 신분증]을 발급합니다.

- 구조(Header.Payload.Signature)
1) Header : 이 시분증이 어떤 알고르짐으로 만들어졌는지 적혀있습니다.
2) Payload : 신분증 주인의 정보(이메일)와 권한(ROLE_USER,ROLE_ADMIN 등)이 적혀있습니다.
3) Signature : 이 신분증이 가짜가 아님을 증명하는 성주(관리자)의 디지털 서명입니다.

- 신분증 발급(createToken) 한 줄 설명
1) Jwts.builder(): 신분증 제작기를 가동합니다.
2) .subject(authentication.getName()) : 신분증 주인 칸에 복호화된 이메일을 적습니다.
3) .claim("auth","ROLE_...") : 이 사람이 일반인인지, 관리자, 변호사인지 권한 배지를 달아줍니다.
4) .signWith(key) : 성주만이 가진 비밀 열쇠로 위조 방지 도장을 쾅 찍습니다. (프로퍼티 설정)


8. 성문의 수문장 : Filter & EntryPoint

- JwtAuthenticationFilter(수문장)
: 모든 방문객의 가방을 뒤져 신분증(JWT)이 있는지 확인합니다.

- JwtAuthenticationEntryPoint(보안관)
: 신분증이 없거나 가짜인 사람이 몰래 들어오려다 걸리면 [로그인이 필요합니다.]라고 외치며 쫓아냅니다.
